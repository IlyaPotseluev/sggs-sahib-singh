# SGGS Sahib Singh / KhojGurbani — техническая документация и концепт

## Назначение

Проект `custom_khoj_sahib_singh` — Python-инструментарий для подготовки русского перевода Sri Guru Granth Sahib на основе данных KhojGurbani и толкования Prof. Sahib Singh. Главный артефакт проекта — набор файлов `ang_json/ang_XXXX.json`; DOCX и будущие приложения/плагины должны собираться из этих JSON, а не становиться единственным источником данных.

Текущая система уже умеет:

- получать строки анга с KhojGurbani API;
- отправлять Gurmukhi + `sahib_singh_pa` в ChatGPT через Playwright;
- сохранять результат в `ang_json`;
- пересобирать DOCX из JSON;
- искать пропущенные/битые/дублирующиеся анги;
- валидировать покрытие против локального `banidb`;
- чинить проблемные анги через Granth Darpan;
- работать через простое интерактивное меню `python khoj.py`.

## Пользовательские истории

- Как читатель я хочу открыть анг SGGS на русском языке, чтобы читать перевод по линии Prof. Sahib Singh без знания панджаби.

- Как администратор я хочу создавать авторов и переводчиков, чтобы у каждого перевода, комментария и размышления была понятная атрибуция.

- Как администратор я хочу защитить исходные импортированные переводы от случайной правки, чтобы не потерять доверенный корпус.
- Как администратор я хочу экспортировать DOCX или PDF по всему SGGS, выбранному ангу или выбранному диапазону, чтобы получать красиво оформленные материалы из админки.
- Как разработчик я хочу иметь стабильный JSON-контракт, чтобы WordPress-плагин, DOCX-сборка и будущие приложения читали один формат.
- Как разработчик я хочу валидировать корпус против `banidb`, чтобы не публиковать анги с пропущенными строками или неверным `verse_id`.
- Как администратор сайта я хочу загрузить готовые JSON в WordPress-плагин, чтобы публиковать SGGS без запуска Python на продакшене.
- Как приложение я хочу загружать единую версию корпуса из API, а при недоступности API использовать последнюю локально сохранённую версию.

## Текущее состояние

### Основные файлы

- `chatgpt_khojgurbani_sahibsingh_bot.py` — основной бот: KhojGurbani API, ChatGPT, JSON, DOCX, scan/retry режимы.
- `khoj.py` — простое главное меню поверх бота и валидатора.
- `validate_angs.py` — проверка `ang_json` против структуры и `banidb`.
- `tests/test_ang_json.py` — pytest-структурные тесты JSON и standalone coverage report.
- `rebuild_from_darpan.py` — восстановление проблемных ангов через `banidb` + Granth Darpan + ChatGPT.
- `ang_json/` — основной корпус данных.
- `shabad_map.json` — вспомогательная карта для ускорения/стабилизации поиска шабадов.
- `khojgurbani_sahibsingh_chatgpt.docx` — генерируемый документ.
- `.progress_khojgurbani_sahibsingh_chatgpt.docx.txt` — progress-файл для продолжения работы.

### Формат ang_json

Текущий базовый контракт:

```json
{
  "ang": 1,
  "line_count": 30,
  "lines": [
    {
      "index": 1,
      "verse_id": 1,
      "shabad_num": 1,
      "shabad_id": 123,
      "gurmukhi": "ੴ ਸਤਿ ਨਾਮੁ ...",
      "site_roman": "ik ōunkār ...",
      "sahib_singh_pa": "ਅਕਾਲ ਪੁਰਖ ਇੱਕ ਹੈ ...",
      "roman": "ik ōankār sat nām ...",
      "translation_ru": "Акал Пуракх един ..."
    }
  ]
}
```

Обязательный runtime-минимум для публикации:

- `ang`;
- `line_count`;
- `lines[].index`;
- `lines[].verse_id`;
- `lines[].gurmukhi`;
- `lines[].roman`;
- `lines[].translation_ru`.

Поля источника, которые желательно сохранять:

- `site_roman`;
- `sahib_singh_pa`;
- `shabad_num`;
- `shabad_id`.

Они нужны для аудита, восстановления и будущего переключения слоёв.

## Архитектурный концепт

### Поток данных

```text
KhojGurbani API
  -> SourceLine[]
  -> ChatGPT prompt
  -> BEGIN_KG_JSON / END_KG_JSON
  -> parse/repair
  -> AngTranslation
  -> ang_json/ang_XXXX.json
  -> validate_angs.py
  -> DOCX / WordPress plugin / будущий API
```

Для восстановления проблем:

```text
banidb verses
  + Granth Darpan page
  -> DARPAN_PROMPT
  -> AngTranslation
  -> ang_json/ang_XXXX.json
```

### Границы ответственности

- Python-бот отвечает за генерацию и восстановление данных.
- `ang_json` является источником истины для публикации.
- DOCX является экспортом, а не источником истины.
- WordPress-плагин должен быть read-only viewer поверх JSON.
- Продакшен-сайт не должен запускать Playwright, ChatGPT или парсинг Darpan по пользовательскому запросу.

### Проверка качества

Сейчас есть два уровня проверки:

1. Быстрые pytest-тесты:
   - JSON читается;
   - `ang` совпадает с именем файла;
   - `line_count == len(lines)`;
   - обязательные поля присутствуют;
   - `verse_id` не дублируется;
   - `gurmukhi` и `roman` не пустые.

2. Coverage/validation против `banidb`:
   - отсутствующие JSON;
   - пустые `lines`;
   - строки из `banidb`, не покрытые нашим JSON;
   - несовпадения Gurmukhi для общих `verse_id`;
   - cross-ang duplicates, когда соседние анги содержат одни и те же `verse_id`.

Рекомендуемый процесс перед публикацией:

```bash
python -m pytest tests/ -v
python validate_angs.py --start 1 --end 1430
```

Если `banidb` локально недоступен, pytest всё равно даёт структурный минимум, но покрытие корпуса проверить нельзя.

## Логи изменений

Для проекта нужны два разных типа логов.

### Подробный технический лог

Технический лог нужен редактору, разработчику и аудитору корпуса. Он может содержать внутренние слова и детали:

- какие анги переведены или пересобраны;
- какие `verse_id` были пропущены и восстановлены;
- какие проблемы нашёл `validate_angs.py`;
- какие строки перепереведены;
- какой источник использовался: KhojGurbani, Sahib Singh, Granth Darpan, `banidb`;
- какие команды запускались;
- какие raw logs сохранены;
- какие known issues остались.

Формат можно вести как Markdown и/или JSONL:

```text
2026-04-25
- Rebuilt ang 1078 and 1079 via Darpan.
- Fixed duplicate verse_id coverage issue.
- Ran pytest tests/ -v: ok.
- Remaining: ang 700 requires manual review.
```

### Публичный лог для приложения

Публичный лог нужен читателю в приложении или на сайте. Он не должен звучать как отчёт программиста.

Правила:

- писать по-русски, коротко и спокойно;
- не использовать слова `API`, `JSON`, `schema`, `checksum`, `Playwright`, `banidb`;
- описывать пользу: стало понятнее, исправлена неточность, добавлены пояснения, расширен словарь;
- не показывать внутренние ошибки, если они уже исправлены;
- группировать изменения по понятным разделам: "Перевод", "Словарь", "Поиск", "Чтение".

Пример:

```text
Что нового

Добавлены новые анги перевода.
Улучшена транслитерация в нескольких строках.
Некоторые пояснения стали короче и яснее.
```

### Связь с пакетами данных

При публикации новой версии корпуса стоит генерировать два артефакта:

```text
release/
  public_update_log.md
  technical_change_log.md
  manifest.json
```

`manifest.json` может ссылаться на оба лога:

```json
{
  "corpus_version": 12,
  "public_update_log": "public_update_log.md",
  "technical_change_log": "technical_change_log.md"
}
```

WordPress-плагин и мобильное приложение показывают только `public_update_log`. Технический лог остаётся для команды и может быть доступен в админке.

## Концепт WordPress-плагина

### Цель

Сделать интерактивный просмотр SGGS на сайте на основе `ang_json`, без базы данных на первом этапе.

### Пользовательские возможности

- открыть конкретный анг по URL;
- переключаться на предыдущий/следующий анг;
- показывать/скрывать Gurmukhi, romanization, русский перевод, исходный Punjabi Sahib Singh;
- скрывать/показывать отдельные строки или слои так же удобно, как в мобильном reader;
- копировать ссылку на строку;
- открывать автора/переводчика и читать краткую биографию;
- видеть комментарии автора к ангу, шабду, паури или строке;
- искать по текущему ангу;
- видеть понятную сводку "Что нового" после обновления корпуса;
- позже — искать по всему корпусу;
- позже — переключать переводчиков и комментарии.

### Сущности

Нужны отдельные сущности, а не только плоский JSON строк.

`Author`:

- `id`;
- `display_name`;
- `slug`;
- `type`: translator, commentator, editor, source_author;
- `bio_short`;
- `bio_full`;
- `website_url`;
- `portrait_id` или media attachment;
- `source_notes`;
- `is_active`.

Примеры:

- Prof. Sahib Singh — источник панджабского толкования;
- Карминдер Сингх Дхиллон — автор KSD-рамки, переводчик/комментатор в соответствующих пакетах;
- будущий русский переводчик или редактор.

`TranslationLayer`:

- `id`;
- `author_id`;
- `language`;
- `title`;
- `description`;
- `source_type`: imported, original, editorial, generated;
- `locked`: защищён ли слой от ручного редактирования;
- `version`;
- `license_notes`.

`Commentary`:

- `id`;
- `author_id`;
- `target_type`: ang, shabad, pauri, line, word;
- `target_ref`;
- `body`;
- `status`: draft, review, published;
- `visibility`: public, private, admins_only;
- `created_by`;
- `updated_by`;
- `version`.

`CorpusVersion`:

- `corpus_version`;
- `schema_version`;
- `content_hash`;
- `published_at`;
- `public_update_log`;
- `technical_change_log`;
- `required_app_min_version`;
- `warning_level`: none, outdated, incompatible.

### Админка

Админка должна быть отдельным рабочим инструментом, а не просто настройками shortcode.

Разделы:

- Corpus: импорт JSON, проверка версии, публикация новой версии.
- Authors: создание авторов, биографии, портреты, ссылки.
- Translations: слои переводов, привязка к автору, lock/protect статус.
- Commentaries: авторские размышления, комментарии к ангу/шабду/паури/строке.
- Exports: генерация DOCX/PDF.
- API/Access: ключи, токены, права приложений.
- Logs: публичный лог и технический лог.
- Settings: кэш, PWA, reader defaults.

Роли:

- `sggs_admin` — полный доступ.
- `sggs_editor` — может редактировать комментарии и новые авторские слои, но не исходный импорт.
- `sggs_author` — может писать свои размышления/комментарии.
- `sggs_reviewer` — может утверждать публикацию.
- `sggs_exporter` — может генерировать DOCX/PDF без права менять корпус.

### Защита исходных переводов

Исходные импортированные слои должны быть защищены от случайной правки:

- imported layers по умолчанию `locked=true`;
- ручная правка locked слоя запрещена без явного "создать редакционную копию";
- любые изменения trusted слоя идут через новую версию или patch overlay;
- перед публикацией показывается предупреждение, если меняется corpus hash;
- админка должна отличать "исходный перевод", "редакционная правка", "авторский комментарий" и "размышление";
- нужна история версий для каждой строки, если появится ручное редактирование;
- массовый импорт не должен перетирать опубликованный слой без backup.

Практическое правило: не ворошить исходный перевод напрямую. Если нужен русский авторский слой — создать нового автора и новый `TranslationLayer`.

### REST API

Минимальный read-only API:

```text
GET /wp-json/sggs/v1/manifest
GET /wp-json/sggs/v1/ang/{number}
GET /wp-json/sggs/v1/search?q={query}
GET /wp-json/sggs/v1/updates
GET /wp-json/sggs/v1/authors
GET /wp-json/sggs/v1/authors/{slug}
GET /wp-json/sggs/v1/commentary/{target_type}/{target_ref}
GET /wp-json/sggs/v1/export/status/{job_id}
```

`manifest` должен возвращать:

- диапазон доступных ангов;
- версию корпуса;
- дату сборки;
- список доступных слоёв;
- ссылку на публичную сводку обновлений;
- checksum, если корпус публикуется как пакет.

`ang/{number}` должен возвращать JSON без преобразования смысла:

- `ang`;
- `line_count`;
- `lines`;
- metadata.

Для приложений нужен единый version contract:

```json
{
  "corpus_version": 12,
  "schema_version": 2,
  "min_supported_app_version": "1.0.0",
  "content_hash": "...",
  "download_url": "...",
  "public_update_log_url": "...",
  "warning": null
}
```

Если приложение не может получить API:

- использовать последнюю успешно сохранённую локальную версию;
- показать мягкое предупреждение: "Показывается сохранённая версия текста";
- не блокировать чтение;
- при восстановлении доступа предложить обновление;
- если версия несовместима, показать явное предупреждение и оставить bundled fallback.

### AJAX, кэш и PWA

Viewer должен быть быстрым:

- navigation по ангам через AJAX/fetch;
- prefetch предыдущего и следующего анга;
- local/session cache на фронтенде для уже открытых ангов;
- server-side cache JSON-ответов через transients/object cache;
- static cache для manifest и public update log;
- ETag/Last-Modified для REST-ответов;
- lazy load комментариев и биографий авторов;
- debounce поиска;
- ограничение результатов поиска;
- PWA service worker как опциональный режим: кэшировать shell, последний manifest, последние открытые анги, reader settings;
- кнопка "обновить сохранённые тексты", если service worker держит старую версию.

PWA не должна становиться источником истины. Источник истины — опубликованный corpus package + manifest.

### Frontend

Первый релиз можно сделать на vanilla JS:

- меньше зависимостей;
- проще подключить в WordPress;
- достаточно для reader UI.

Если интерфейс станет сложнее, можно перейти на Vue/React, но не раньше появления:

- глобального поиска;
- пользовательских настроек;
- словарных popup;
- нескольких переводчиков/комментаторов.

Reader controls:

- toggles для Gurmukhi, roman, русский перевод, Punjabi source, комментарии;
- "compact mode" для плотного чтения;
- hide/show для строк или блоков, чтобы пользователь мог временно убрать лишние слои;
- сохранение настроек в localStorage;
- deep links вида `?ang=1#line-123`;
- состояние загрузки для AJAX без дёргания интерфейса;
- понятные предупреждения при устаревшем корпусе.

### Безопасность и производительность

- API только читает файлы из разрешённой папки.
- Номер анга валидируется как integer `1..1430`.
- Поиск по файлам должен иметь лимиты по длине запроса и количеству результатов.
- JSON-ответы кэшируются WordPress transients или static cache.
- На первом этапе не нужна база данных.
- Никаких LLM/API ключей в WordPress-плагине.
- Публичный экран обновлений читает только пользовательский лог, не технический.
- Admin REST endpoints требуют WordPress nonce и capability checks.
- Public REST endpoints не должны раскрывать технические логи, raw prompts или внутренние пути файлов.
- API-ключи для приложений хранить в WordPress options/table в зашифрованном или хотя бы hashed виде, где возможно.
- Не показывать секреты повторно после создания; давать rotate/revoke.
- Для мобильных приложений лучше использовать revocable app tokens с ограниченными правами read-only.
- Добавить rate limit для public search и export endpoints.
- Экспорт DOCX/PDF запускать как background job, не в web request.
- Загружаемые JSON-пакеты проверять по hash и версии схемы.
- Перед импортом новой версии делать backup текущего manifest и corpus index.

### Ключи доступа и отдельные таблицы

Если сайт станет источником данных для приложений, нужны управляемые ключи:

- отдельная таблица `wp_sggs_api_keys` или custom post type не подходит для секретов;
- хранить `key_id`, hashed secret, label, scopes, created_at, last_used_at, revoked_at;
- scopes: `read_manifest`, `read_content`, `read_updates`, `export_status`;
- ключ показывается только один раз при создании;
- rotate без потери доступа: создать новый, проверить приложение, отозвать старый;
- emergency revoke all;
- audit log использования ключа без хранения полного секрета.

### Экспорт DOCX/PDF

Админка должна уметь собрать красиво оформленный экспорт:

- весь SGGS;
- выбранный анг;
- диапазон ангов;
- выбранный шабд/паури, если будет стабильная структура;
- выбранные слои: Gurmukhi, roman, русский перевод, Punjabi source, комментарии;
- выбранные авторы/переводчики;
- титульная страница с версией корпуса;
- оглавление;
- цвета как в текущем DOCX;
- footer с ang/страницей/версией;
- предупреждение, если выбранный экспорт содержит draft/commentary не опубликованный для сайта.

DOCX можно генерировать сервером, PDF — либо через headless renderer, либо через DOCX->PDF pipeline, но как background job. Результат хранить временно и удалять по расписанию.

### Единая версия для сайта и приложений

Сайт, Nitnem app и будущие SGGS-приложения должны читать одну версию корпуса:

- manifest — центральный контракт;
- corpus package — единый артефакт;
- приложения кешируют последнюю успешную версию;
- сайт показывает текущую опубликованную версию;
- админка предупреждает, если новая версия требует обновления приложения;
- приложение предупреждает, если локальная версия старая;
- несовместимая схема не должна молча загружаться;
- rollback должен быть возможен через публикацию предыдущего manifest.

## Необходимые знания Python для разработки с нуля

Минимальный уровень:

- Python 3.11+ или совместимая версия;
- виртуальное окружение: `python -m venv .venv`;
- установка зависимостей через `pip`;
- `argparse` для CLI-флагов;
- `dataclasses` для структур `SourceLine`, `OutputLine`, `AngTranslation`, `RuntimeConfig`;
- `pathlib.Path` для путей;
- `json` и UTF-8 для `ang_json`;
- `urllib.request` или `requests` для API;
- `playwright` для браузерной автоматизации;
- `python-docx` для сборки DOCX;
- `sqlite3` для чтения `banidb` и логов валидации;
- `pytest` для тестов;
- регулярные выражения `re` для извлечения JSON и нормализации;
- `unicodedata.normalize` для аккуратной работы с Gurmukhi.

Навыки, критичные именно здесь:

- писать идемпотентные скрипты, которые можно безопасно перезапускать;
- сохранять частичные результаты после каждого анга;
- проектировать JSON как публичный контракт;
- отличать source fields от generated fields;
- логировать raw prompts/responses для аудита;
- делать repair-path для невалидного ответа LLM;
- ограничивать контекст промпта, чтобы не переполнить модель;
- валидировать результат до записи или перед публикацией;
- не смешивать генерацию данных и отображение данных.
